# Backlog (Betreiber-/Entwicklungs-Anforderungen)

Von der Referenzanlage bzw. im Entwicklungsprozess aufgekommene Anforderungen, die nicht
aus dem Community-Katalog stammen. Kandidaten für Priorisierung in den jeweiligen Phasen.

| # | Anforderung | Quelle | Warum / Kontext | Zielphase |
|---|---|---|---|---|
| B-1 | **Partielle Aktivierung: Logik und Visu getrennt aktivieren, ohne kompletten Neustart.** | Betreiber, 09.07.2026 | In EDOMI erzwingt jede Projektaktivierung einen **kompletten Neustart** der Runtime (und je nach Einstellung sogar einen Reboot). Für iteratives Arbeiten — gerade agentisch — ist das teuer: Jede kleine Logik-/Visu-Änderung kostet einen vollen Hochlauf. Fachwerk sollte Änderungen an einzelnen Logikseiten bzw. Visuseiten **inkrementell und heiß** aktivieren können (Hot-Reload je Artefakt), ohne KO-Zustände, Timer und Bus-Verbindung zu verlieren. Passt direkt zum Agent-first-Ziel (schneller Edit-Test-Zyklus) und zur EDOMI-Stärke „Live-Projektierung", die Fachwerk noch feinkörniger machen soll. | Architektur (Phase 2, ADR) → MVP/Phase 3+ |
| B-2 | **Historie / Versionsverwaltung für Logiken und Visus** (Änderungsverlauf, Diff, Rollback, „wer/wann/was"). | Betreiber, 09.07.2026 | Fehlte in EDOMI durchgehend: Änderungen an Logik/Visu sind nicht nachvollziehbar, kein Zurückrollen auf einen früheren Stand, kein Vergleich zweier Versionen. **Fachwerk bekommt das strukturell fast geschenkt**, weil Projekte = Textdateien in Git sind (Plan § 4.1, Agent-first): `git log`/`diff`/`blame`/`revert`/Branches liefern Historie, Vergleich, Schuld und Rollback nativ. Aufgabe ist v. a. die **UX-Schicht**: Verlauf/Diff/Restore direkt in der Editor-Oberfläche sichtbar machen (visuelle Diffs für Visu-Seiten und Logikgraphen), damit Nicht-Git-Nutzer es bedienen können — die Engine dahinter ist Git. Verbindet sich mit B-1 (jede Aktivierung = ein benennbarer Commit) und mit dem Provenance-/Trace-Gedanken. | Architektur (Phase 2) → UI (Phase 5) |

| B-3 | **Direkte Manipulation / Drag-and-Drop in Logik- und Visueditor** (Elemente, Bausteine, KOs per Ziehen einfügen/verbinden). | Betreiber, 09.07.2026 | In EDOMI läuft im Logik- **und** Visueditor praktisch alles über **Rechtsklick/Kontextmenü** — es gibt kein Drag-and-Drop, um Visuelemente, Logikbausteine oder KOs einzufügen oder zu verbinden. Das ist zäh und für Einsteiger abschreckend, obwohl der pixelgenaue No-Code-Editor eigentlich EDOMIs ★★★-USP ist (Community-Katalog). Fachwerk-Editoren sollen **direkte Manipulation** bieten: Elemente/Bausteine aus einer Palette ziehen, Verbindungen ziehen, KOs per Drag zuweisen, Mehrfachauswahl, Ausrichten/Snapping. Wichtig: Der Editor bleibt **Ansicht auf das Textprojekt** (Plan § 4.1) — Drag-and-Drop erzeugt/ändert die deklarativen Dateien, sodass Mensch (Maus) und Agent (Text/API) dasselbe Modell bearbeiten. | UI (Phase 4 Logik / Phase 5 Visu) |

| B-4 | **Anwesenheitssimulation: einfach deklarativ statt nur Record/Replay.** | Betreiber, 10.07.2026 | EDOMI kann nur „1 Woche Bus-Traffic aufzeichnen und (gefiltert auf KOs) abspielen". Gewünscht ist zusätzlich ein **einfacher, deklarativer Ferienmodus** im Stil der AVM-Heizungs-Feriensteuerung: „Licht an 18–22 Uhr", „Rollo hoch/runter um X", mit leichter Zufallsstreuung — schnell konfiguriert statt erst eine Woche aufzeichnen. Fachwerk-Ansatz: (a) deklarative Szenario-Datei im Gewerk (ADR-0004-Format, damit Editor & Agent sie bauen können; UI im AVM-Stil), (b) Record/Replay als zweiter Modus — die Grundlagen existieren schon als Bus-Simulator M3/M4 (Mitschnitt/Replay, SPEC-008); dieselbe Mechanik kann als „Anwesenheits-Playback" in den Core wandern. | Fachlogik/Bausteine (Phase 4/5) |

| B-5 | **DEV- und PROD-Gewerk als Standard-Arbeitsmodell** („Umgebungen"): gemütlich auf DEV bauen und erst nach PROD überführen, wenn alles passt — ODER bewusst kleine Änderungen direkt in PROD. | Betreiber, 10.07.2026 | EDOMI kennt zwar mehrere „Projekte" mit Aktivierungs-Wechsel, aber keinen echten DEV→PROD-Workflow. Fachwerk hat die Bausteine strukturell schon: Gewerk = Git (ADR-0004) ⇒ **DEV = Branch/Worktree oder zweite Instanz**, „nach PROD pushen" = Merge + Aktivierung (B-1) mit Diff-Review davor (B-2); der Bus-Simulator (SPEC-008 M2/M3) gibt der DEV-Instanz eine gefahrlose Bus-Gegenstelle bis hin zum Schattenbetrieb mit Echtdaten. Zu entwerfen ist der **einfache, geführte Workflow** obendrauf: „Umgebung: DEV/PROD"-Schalter, ein-Klick-Promote mit Diff-Ansicht, Direkt-Edit in PROD bleibt erlaubt (mit Hinweis + auto-Rücksync nach DEV, damit die Stände nicht divergieren). Passt zudem exakt zum Human-in-the-loop-Deploy der Agenten-Guardrails (Plan § 4.2). | Architektur-Konzept Phase 2/3, UX ab Phase 5 |

| B-6 | **Baustein-Editor im Browser (Null-Toolchain) + „Volle Regale"-Programm.** | ADR-0008-Review, 10.07.2026 | Adoptions-kritisch (ADR-0008 S-7): Baustein-Autoren dürfen **keine** lokale Toolchain brauchen (kein Node/npm/tsc) — eingebauter Editor mit Manifest-Formular, Live-Test gegen Simulator, Ein-Datei-Einfachheit wie EDOMI-LBS. Dazu vor Community-Launch ein Portierungsprogramm: Standardbibliothek + die meistgenutzten Community-LBS (agentengestützt portiert, Autoren-Zustimmung gem. Plan § 3.4) nativ verfügbar, damit der Baustein-Store zum Start gefüllt ist. **Konkreter erster Schritt — Nachfrage-Katalog:** Forum-/Downloadbereich des KNX-Userforums scannen und einen Katalog erstellen: welche LBS existieren, welche werden meistgenutzt/-gewünscht, welche Kategorien (Fachlogik vs. Fremdsystem-Anbindung). **Rechtliche Leitplanke:** Der Scan liefert *Metadaten und Priorisierung* — niemals LBS-Code zur Übernahme. Umsetzung je Baustein über die drei sauberen Wege: (a) generische Funktion → eigene Standardbibliothek, (b) Autoren-Zustimmung zur Portierung, (c) Clean-Reimplementierung der *Funktionalität* aus Verhaltensbeschreibung. Fremdsystem-LBS (Sonos, Hue, …) werden ohnehin als Treiber/Service-Bausteine gegen die APIs der Systeme neu gebaut — null EDOMI-Bezug. | SDK/Editor Phase 4, Katalog ab sofort möglich, Programm Phase 6/7 |

| B-7 | **Animierte Visu-Elemente** (weiche Übergänge: Schalter-Toggle gleitet, Werte blenden/zählen, Popups faden, Zustandswechsel je Design animierbar). | Betreiber, 19.07.2026 | In EDOMI praktisch unmöglich — der Betreiber hat animierte Switches dort vergeblich versucht; designtechnisch aber genau das, was moderne Visus ausmacht. Fachwerk bekommt das fast geschenkt: Elemente sind DOM+CSS, `design_je_wert` (R-9) wechselt Klassen — CSS-Transitions/Keyframes darauf sind ein kleines Renderer-/Design-Schema-Feature (z. B. `uebergang: {dauer_ms, kurve}` in der Design-Vorlage). Wichtig: deklarativ im Visu-Format (ADR-0004/P5-6), damit Editor & Agent es setzen können; sparsame Defaults (Barrierefreiheit: `prefers-reduced-motion` respektieren). | Visu-Renderer (P5-UI-B/Phase 5), Editor-Support P5-10 |
| B-8 | **Statischer Text auf Visu-Elementen** (freies Textfeld für `label`/Presets). | P5-9-Import, 21.07.2026 | Beim Visu-Import (Stufe 3) fiel auf: das P5-6-Schema hat KEIN Textfeld für Preset-Elemente (nur `widget` trägt `parameter`). Statische Beschriftungen des Betreibers (Raumnamen, Überschriften — allein im Referenz-Gewerk 48 Stück) haben deshalb keinen Platz und landen aktuell nur in `seite.notizen` + Import-Report. Lösung: ein deklaratives Textfeld für Presets (z. B. `text:` am Element oder ein `label`-spezifischer `parameter.text` — Schema-Entscheidung/ADR-Notiz), Renderer + Editor + Visu-Importer ziehen nach. Klein und mit klarem Nutzen (Import wird sofort lesbarer). | Schema-Änderung (Spur 1) + Renderer/Editor (Spur 2) + Importer-Nachzug (P5-9) |
